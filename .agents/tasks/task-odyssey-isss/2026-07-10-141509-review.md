# ODYSSEY ISSS v2: Fixes for auth, rate limiting, appointment safety, and CSS

Six issues from the first review have been addressed: the staff registration secret is now read from `STAFF_REGISTRATION_CODE` env var (no longer in source), the global CSS `*` transition rule is removed, the AI model is `gpt-4o`, rate limiting (token-bucket) is applied to both auth registration and chat, the appointment booking uses a Prisma `$transaction` to prevent double-booking, and `NEXTAUTH_SECRET` is validated at startup against a list of weak values.

**Watch for:** Rate limiting on login/authorize is not possible through the current NextAuth credentials flow — only the registration tRPC mutation is rate-limited (confirmed). The in-memory rate limiter resets on every serverless cold start or hot reload, making it ineffective in production (confirmed). Notification `markAsRead` lacks ownership verification — any authenticated user can mark another user's notification as read by guessing the ID (confirmed). The `bulkSetAvailability` mutation runs N sequential DB round-trips without a transaction (likely).

## High-level view

The auth posture is substantially improved. Staff registration validates a server-only env var and fails closed if the var is missing. However, login brute-force protection remains absent because NextAuth's `authorize` callback has no hook point for the tRPC rate limiter — the rate limit only applies to `auth.register`, not to the actual credential verification path through `/api/auth/[...nextauth]`.

The rate limiter is architecturally limited: as an in-memory `Map`, it shares no state across serverless function instances and resets on every cold start. For the stated capstone demo scope this is acceptable; the code documents the Redis migration path clearly. But the login path it should protect most is not wired up at all.

The notification router exposes `markAsRead` without verifying the notification belongs to the calling user. `markAsRead` accepts an arbitrary notification ID — if IDs are guessable or leak in any response, one user can silence another's notifications. `markAllAsRead` correctly scopes to the current user, making the asymmetry apparent.

The `bulkSetAvailability` mutation runs a sequential find+create/update loop per date without wrapping the batch in a transaction, risking partial state on failure or concurrent access.

## Login path remains unthrottled

The `authRateLimiter` is invoked only in the tRPC `auth.register` mutation:

```typescript
const rateLimitResult = authRateLimiter.check(`register:${input.email}`);
```

The actual login flow goes through NextAuth's `authorize` callback in `src/server/auth.ts`, which performs `bcrypt.compare` directly with no rate check:

```typescript
async authorize(credentials) {
  if (!credentials?.email || !credentials?.password) return null;
  const user = await db.user.findUnique({ where: { email: credentials.email } });
  if (!user) return null;
  const isPasswordValid = await bcrypt.compare(credentials.password, user.passwordHash);
  if (!isPasswordValid) return null;
  return { id: user.id, ... };
}
```

An attacker can POST to `/api/auth/callback/credentials` without any throttling. The rate limiter infrastructure exists — it just needs to be called here. Import `authRateLimiter` and check `authRateLimiter.check(\`login:${credentials.email}\`)` at the top of `authorize`, returning null when the limit is exceeded.

## Notification ownership gap

In `src/server/trpc/routers/notification.ts`, `markAsRead` does:

```typescript
markAsRead: protectedProcedure
  .input(z.object({ id: z.string() }))
  .mutation(async ({ ctx, input }) => {
    await ctx.db.notification.update({
      where: { id: input.id },
      data: { isRead: true },
    });
    return { success: true };
  }),
```

No check confirms `notification.userId === ctx.session.user.id`. The fix is to add a compound where filter: `{ id: input.id, userId: ctx.session.user.id }`. This makes the update a no-op if the notification belongs to someone else, matching the pattern `markAllAsRead` already uses.

## Rate limiter cold-start behavior

The `RateLimiter` class in `src/lib/rate-limit.ts` stores state in a `Map`:

```typescript
private store: Map<string, RateLimitEntry> = new Map();
```

In Vercel/serverless deployments, each function invocation may get a fresh process. Even in long-running Node, a deploy or crash resets all limits. The fundamental issue: rate limits that matter (brute-force protection) cannot rely on in-process memory in production. The code's comment recommending `@upstash/ratelimit` is the correct production path, but for this to be an effective demo, at minimum the login path needs to be wired up so the pattern is demonstrable.
