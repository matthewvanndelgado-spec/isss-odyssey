# ODYSSEY ISSS: Greenfield Next.js Application for University of Batangas

A full-stack capstone application providing international student services (visa tracking, appointment scheduling, inquiry system, exchange programs, AI assistant) built on Next.js 14 App Router with tRPC v10, Prisma/SQLite, NextAuth.js v4, and shadcn/ui. The architecture follows a clean role-based pattern (Student/Staff) with a tRPC layer enforcing auth at the procedure level. The implementation covers ~21,000 lines across 99 files and produces 29 working routes.

**Watch for:** Hardcoded staff registration code in source (confirmed). No rate limiting on auth or AI chat endpoints despite being listed as a requirement (confirmed). NEXTAUTH_SECRET has no entropy enforcement and the `.env.example` ships a placeholder (confirmed). Document upload accepts arbitrary URLs with no S3 integration despite the spec requiring pre-signed URLs (confirmed). The global CSS `transition-property` rule on `*` will fight with Tailwind animations and create performance overhead on large DOMs (likely).

## High-level view

The authentication system uses NextAuth.js credentials with JWT sessions and bcrypt hashing, gated by middleware that checks routes and roles. Staff registration is protected only by a hardcoded secret (`ISSO-STAFF-2024`) embedded in the source code of the auth router — it ships in the client bundle. No rate limiting exists on the login or register endpoints, and no account lockout or failed-attempt tracking is present.

The tRPC API surface is well-layered: `publicProcedure`, `protectedProcedure`, and `staffProcedure` enforce access control at the middleware level, and Zod schemas validate all inputs. The router topology covers 10 sub-routers with cursor-based pagination, conflict detection on appointments, and owner-scoped data access. The ownership check pattern (`if not staff and not owner, throw FORBIDDEN`) is repeated manually in each router that needs it — no abstraction prevents a developer from forgetting the check on a new endpoint.

The AI assistant uses OpenAI GPT-4o-mini (not GPT-4o as spec'd) with a keyword-based RAG pipeline against OrientationContent. There is no streaming implementation despite the spec requiring "AI chat responses < 3 seconds with streaming." For a 1024-token response, typical non-streaming completion time is 3-8 seconds.

The Prisma schema uses SQLite (not PostgreSQL with pgvector as spec'd) with string-typed enums, JSON-as-string fields for time slots, and no encryption at rest. This is a deliberate dev-mode choice documented in the RAG module comments, but means the production requirements (PII encrypted at rest, pgvector embeddings) are entirely unimplemented.

The frontend uses `"use client"` pages throughout with tRPC hooks, shadcn/ui components, and role-based conditional rendering. The UB brand colors are correctly mapped to CSS custom properties. A global `*` transition rule in the CSS will interfere with Tailwind animations and add overhead on element-heavy pages.

1. **Hardcoded staff registration secret** — The code `ISSO-STAFF-2024` is in `src/server/trpc/routers/auth.ts`. Anyone reading the client bundle or source can register as staff. Move to an environment variable and validate server-side only.
2. **No rate limiting on auth endpoints** — The spec requires rate limiting on auth and AI chat. Neither `login` (NextAuth authorize) nor `register` (tRPC mutation) have any throttling. Add rate limiting middleware (e.g., `upstash/ratelimit` or express-rate-limit equivalent).
3. **No rate limiting on AI chat** — `chat.sendMessage` has no request throttle. A single user can exhaust OpenAI quota. Add per-user rate limiting.
4. **Document upload accepts raw URLs** — `visa.addDocument` accepts any `z.string().url()` and stores it as `fileUrl`. No S3 pre-signed URL generation, no file upload flow, no expiration. Placeholder that exposes the system to SSRF if URLs are ever fetched server-side.
5. **NEXTAUTH_SECRET placeholder** — `.env.example` contains `"your-secret-key-here"`. No validation prevents the app from running with this weak secret. Add a startup check for minimum secret length/entropy.
6. **AI model mismatch** — Code uses `gpt-4o-mini`; spec requires `gpt-4o`. Either update the model or document the trade-off.
7. **No streaming for AI responses** — Spec requires streaming chat responses < 3s. Current implementation awaits the full completion synchronously via tRPC mutation. Switch to an SSE/ReadableStream endpoint.
8. **Appointment booking race condition** — No transaction wraps the conflict-check-then-insert in `appointment.create`. On PostgreSQL (production), concurrent requests can double-book a slot. Add a unique compound index or use a serializable transaction.
9. **SQLite instead of PostgreSQL/pgvector** — All production data requirements (PII encryption, vector search, concurrent write performance) are unmet. Acceptable for capstone demo but must be documented as a limitation.
10. **Global CSS transition on all elements** — `* { transition-property: color, background-color... }` in `globals.css` causes unnecessary composite layer creation on large pages and can conflict with Tailwind's animation utilities.

## Staff registration secret and auth posture

The staff registration flow in `src/server/trpc/routers/auth.ts` uses a Zod refine to check `data.staffCode !== "ISSO-STAFF-2024"`. This string is compiled into the client-side JavaScript bundle because the entire validation schema (including the refine) runs client-side for error display. Any user inspecting the bundle or reading the repository can self-register as staff with full administrative access.

```typescript
}).refine((data) => {
  if (data.role === "STAFF" && data.staffCode !== "ISSO-STAFF-2024") {
    return false;
  }
  return true;
}, ...);
```

The fix is to move the staff code to a server-only environment variable and validate it only in the mutation handler, not in the Zod schema that ships to the client. Alternatively, remove self-registration for staff entirely and seed staff accounts through an admin flow.

There is no account lockout, no failed-attempt tracking, and no CAPTCHA integration on the login flow. Without rate limiting, an attacker can brute-force credentials without friction.

## tRPC authorization and the ownership check pattern

The `visa.addDocument` mutation allows any authenticated user to add documents to a visa record, gated by an ownership check:

```typescript
if (ctx.session.user.role !== "STAFF" && record.studentId !== ctx.session.user.id) {
  throw new TRPCError({ code: "FORBIDDEN" });
}
```

The concern is that `visaRecordId` is a CUID — not guessable in isolation, but if record IDs ever appear in URLs, query params, or error messages, any student could add documents to another student's visa record. The same pattern appears in `visa.getById`, `appointment.getById`, and `inquiry.getById`. A `studentOwnedProcedure` middleware that automatically scopes queries to the current user's records would prevent accidental omission of this check on future endpoints.

## AI assistant: model, RAG, and missing streaming

The OpenAI integration uses `gpt-4o-mini` rather than the specified `gpt-4o`. The `generateChatResponse` function awaits the full completion without streaming:

```typescript
const response = await client.chat.completions.create({
  model: "gpt-4o-mini",
  messages: [...],
  max_tokens: 1024,
  temperature: 0.7,
});
```

For a 1024-token response, typical completion time is 3-8 seconds — exceeding the spec's 3-second requirement. The fix is to use `stream: true` and return chunks via a ReadableStream or SSE endpoint rather than a tRPC mutation.

The RAG pipeline (`src/lib/ai/rag.ts`) loads all `OrientationContent` into memory on every query and performs keyword scoring. For the current demo dataset (~15 documents) this is fine, but will degrade linearly with content growth. The comments correctly document the pgvector migration path.

The chat router stores the user message *before* generating the AI response. If the database write for the assistant response fails after OpenAI succeeds, the response is lost with no retry mechanism.

## Database schema: race conditions and missing constraints

The schema uses SQLite with string-typed enums (no DB-level enforcement), JSON stored as strings in `StaffAvailability.timeSlots`, and CUID primary keys.

- No composite unique constraint on `(programId, studentId)` in `ExchangeProgramApplication` — the application-level check for duplicates is present but a race condition could create duplicates under concurrent requests.
- No cascade delete rules — `Inquiry.delete` manually deletes responses first. If additional relations were added, orphans would appear.
- `VisaDocument.fileUrl` stores raw paths like `/uploads/passport_kim_minjun.pdf` in the seed, but there's no upload handler or file serving route. These URLs are non-functional.

## Appointment booking race condition

The appointment creation checks for staff slot conflicts and student time conflicts *then* inserts. There's no transaction wrapping these operations:

```
1. findFirst(...) → no conflict found
2. findFirst(...) → no student conflict
3. create(...) → appointment created
```

Under concurrent requests on PostgreSQL, two students could both pass step 1 and both create appointments for the same slot. SQLite's single-writer model masks this in development. A unique compound index on `(staffId, date, timeSlot)` with a partial filter for active statuses, or a serializable transaction, would prevent double-booking.

## Global CSS transition performance

The rule in `globals.css`:

```css
* {
  transition-property: color, background-color, border-color, outline-color;
  transition-timing-function: cubic-bezier(0.4, 0, 0.2, 1);
  transition-duration: 150ms;
}
```

This applies to every DOM element, causing the browser to track transition state for all color-related properties on every node. On pages with hundreds of elements (inquiry lists, booking calendars), this adds layout thrashing. It also conflicts with Tailwind's `animate-bounce`, `animate-spin`, and other utilities that set their own `transition-property`. Replace with targeted `transition-colors` classes on interactive elements.

## Missing spec requirements

Several non-functional requirements from the spec are entirely absent:

- **Rate limiting**: No implementation on any endpoint.
- **Pre-signed S3 URLs**: No AWS SDK integration, no upload handler, no URL generation.
- **AWS SES**: No email sending implementation.
- **PII encryption at rest**: No encryption layer.
- **WCAG 2.1 AA**: Partial — form inputs have labels and aria attributes, but no skip navigation, no focus management beyond Radix defaults, no high-contrast mode.
